# CLAUDE.md

## Важно: это НЕ обычный C#

Этот проект работает в режиме `NativeAOT + NoStdLib` и таргетит `UEFI` (`EFI_APPLICATION`), поэтому это не классический .NET-приложение.

Ключевые ограничения:
- Нет обычной BCL в привычном виде (`NoStdLib=true`).
- Базовые типы `System` и runtime-заглушки определяются вручную.
- Точка входа экспортируется как `EfiMain` через `RuntimeExport`.
- Код работает с UEFI ABI и указателями (`unsafe`, `EFI_SYSTEM_TABLE*`).
- Это low-level среда; многие обычные .NET-паттерны тут неприменимы.

Практическое правило:
- Не писать код как для обычного `dotnet console`.
- Все платформенные детали держать в `Boot/Hal`, бизнес-логику в `Kernel/TestApp`.

## Стратегия: постепенный переход в managed

Ключевая долгосрочная цель — **расширять `std/` и уходить в managed C# по максимуму**.

- Любая новая системная утилита (строки, коллекции, арифметика) сначала появляется в `std/no-runtime/shared/`, а не в виде inline-кода внутри ядра или SDK.
- `unsafe`-код и ручные указатели — только там, где иначе нельзя (ABI-граница, шеллкод, прямой доступ к железу).
- Если что-то уже есть в BCL и его можно реализовать без нарушения NoStdLib-контракта — реализовать в `std/` по тому же API (PadLeft, PadRight, Concat…).
- По мере роста `std/` — убирать дублирующий низкоуровневый код из слоёв ядра и SDK.

### Рабочий порядок при реализации фичи

Когда нужна новая строковая/numeric/utility операция:

1. **Сначала — в `std/no-runtime/shared/`.** Написать реализацию как managed C#, с публичным API идентичным BCL (например `string.Substring`, `char.IsDigit`).
2. **Потом — использовать в ядре и SDK.** Existing kernel/SDK код, который делал то же самое inline с unsafe/stackalloc, мигрируется на вызов `std/`.
3. **Inline unsafe оставлять только там, где managed реально нельзя.** Пример: `Console.*Raw` остаётся на stackalloc, потому что `HeapDiagnostics` не может аллоцировать во время обхода блоков кучи. Но такие случаи — исключения, а не норма.

Цель: чтобы обычный C# код из "мира `std/`" выглядел и работал как обычный C# в обычном .NET. Всё что выходит за рамки (unsafe, платформенные детали) — локализовано и аргументировано.
